# 스터디모임 트러블슈팅

구현 과정에서 단순 기능 추가보다 사용자 흐름과 데이터 일관성에 영향을 주었던 문제를 정리했습니다.

## 1. 로그인 사용자와 화면 상태가 어긋나는 문제

### 문제

로그인 직후에도 글쓰기나 반응 기능이 비로그인 상태로 판단되거나, 다른 계정의 캐릭터 설정이 그대로 보이는 문제가 있었습니다.

### 원인

- 프런트의 현재 사용자 상태와 서버 세션 확인 시점이 달랐습니다.
- 캐릭터 설정을 사용자 구분 없이 하나의 로컬 저장 키에 보관했습니다.

### 해결

- 로그인·로그아웃 후 `/api/users/me`를 다시 조회하고 관련 UI를 한 경로에서 갱신했습니다.
- 캐릭터 저장 키에 사용자 ID를 포함해 계정별 설정을 분리했습니다.
- 로그인 필요 기능은 API 요청 전에 공통 로그인 유도 흐름을 거치도록 정리했습니다.

### 배운 점

세션 인증에서는 로그인 API 성공만으로 화면 상태가 완성되지 않습니다. 서버의 현재 사용자 정보와 프런트 상태를 다시 동기화하는 과정이 필요했습니다.

## 2. 조회수와 반응 수의 중복 증가

### 문제

상세 내용을 다시 렌더링하거나 버튼을 빠르게 연속 클릭하면 조회수와 반응 수가 사용자의 의도보다 많이 변경될 수 있었습니다.

### 해결

- 조회한 모집글 ID를 세션에 기록해 같은 브라우저 세션에서는 한 번만 증가시켰습니다.
- 화면 갱신 목적의 상세 요청에는 `increaseView=false`를 사용했습니다.
- 좋아요·북마크·모집 상태 버튼은 요청 처리 중 다시 누를 수 없도록 잠그고 짧은 쿨타임을 적용했습니다.
- DB에는 좋아요와 북마크의 사용자·모집글 복합 unique 제약을 두었습니다.

### 배운 점

UI의 클릭 방지는 사용성을 위한 장치이고, 최종 데이터 중복 방지는 서버와 DB가 담당해야 합니다.

## 3. 답글 등록 후 바로 보이지 않는 문제

### 문제

답글은 정상 저장됐지만 사용자가 현재 스크롤 위치에서는 새 답글을 확인할 수 없었습니다.

### 해결

- 등록 성공 후 댓글 목록을 다시 요청했습니다.
- 새로 생성된 답글 ID를 기준으로 해당 요소까지 스크롤하고 잠시 강조했습니다.
- 댓글 수정 시에도 기존 내용을 편집 입력란에 먼저 채워 사용자가 맥락을 잃지 않도록 했습니다.

### 배운 점

API 성공과 사용자에게 성공이 전달되는 것은 다릅니다. 변경 결과가 즉시 보이는 것까지 하나의 기능 흐름으로 봐야 했습니다.

## 4. 상세 모달 스크롤이 배경 페이지까지 전달되는 문제

### 문제

모집글 상세 모달 내부에서 끝까지 스크롤하면 뒤의 메인 페이지가 함께 움직였습니다.

### 해결

- 모달이 열린 동안 문서 본문의 스크롤을 잠갔습니다.
- 상세 카드에 독립적인 최대 높이와 `overflow`를 적용했습니다.
- 모달 종료 시 기존 스크롤 상태를 복구했습니다.

### 배운 점

모달은 시각적으로 위에 표시하는 것뿐 아니라 포커스와 스크롤 범위도 배경 화면과 분리해야 합니다.

## 5. 관리자 목록의 데이터량과 작업 동선

### 문제

회원, 모집글, 댓글을 한 화면에 모두 출력하면 데이터가 늘어날수록 찾기 어렵고 삭제 버튼까지 가로 스크롤해야 했습니다.

### 해결

- 관리 대상을 탭으로 분리했습니다.
- 회원·모집글·댓글·신고에 검색 조건을 추가했습니다.
- 모집글은 행 클릭 시 미리보기 모달을 열고 그 안에서 삭제하도록 변경했습니다.
- 삭제·제재·신고 처리 결과를 전용 알림으로 표시하고 목록을 자동 갱신했습니다.
- 관리자 작업을 `admin_action_logs`에 남겨 처리된 작업을 다시 확인할 수 있게 했습니다.

### 배운 점

관리자 화면은 데이터 표시량보다 검색과 판단, 실행 후 확인까지의 동선을 먼저 설계해야 했습니다.

## 6. 비밀번호 재설정의 본인 확인 보강

### 문제

이메일과 닉네임만 알면 새 비밀번호를 입력할 수 있는 초기 구조는 실제 서비스의 본인 확인으로 부족했습니다.

### 해결

- 이메일로 6자리 인증번호를 전송하도록 변경했습니다.
- 인증번호는 원문 대신 해시로 저장하고 5분 만료와 실패 횟수 제한을 적용했습니다.
- 인증 성공 후 10분 동안 한 번만 사용할 수 있는 재설정 토큰을 발급했습니다.
- 로컬 개발에서는 메일 발송을 끄고 인증번호를 서버 콘솔에서 확인할 수 있도록 분리했습니다.

### 배운 점

민감한 작업은 사용자 정보 확인과 실제 소유 확인을 구분해야 하며, 인증번호와 토큰에도 만료와 재사용 방지가 필요했습니다.

## 7. 온라인 멤버가 실제 접속 상태가 아니었던 문제

### 문제

초기 우측 멤버 목록은 현재 사용자와 최근 댓글 작성자를 조합한 표시라 실제 온라인 상태를 의미하지 않았습니다.

### 해결

- 세션 인증을 확인하는 WebSocket `/ws/presence`를 추가했습니다.
- 연결과 해제 시 온라인 멤버 목록을 갱신했습니다.
- 같은 사용자가 여러 탭을 열어도 사용자 단위로 중복 표시되지 않게 처리했습니다.

### 배운 점

실시간 상태는 기존 DB 데이터의 조합으로 추측하기보다 연결 생명주기를 직접 관리해야 정확하게 표현할 수 있습니다.
