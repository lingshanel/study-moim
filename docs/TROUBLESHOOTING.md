# 트러블슈팅

구현 과정에서 단순 기능 추가보다 사용자 흐름과 데이터 일관성에 영향을 주었던 문제를 정리했습니다.

## 1. 로그인 후에도 화면 상태가 바뀌지 않는 문제

### 문제

로그인 직후에도 글쓰기나 반응 기능이 비로그인 상태처럼 보이거나, 다른 계정의 캐릭터 설정이 그대로 보이는 문제가 있었습니다.

### 해결

- 로그인/로그아웃 후 `/api/users/me`를 다시 호출해 현재 사용자 정보를 동기화했습니다.
- 캐릭터 설정을 사용자 ID 기준으로 분리해 계정별로 저장되도록 변경했습니다.
- 로그인 필요 기능은 API 요청 전에 공통 로그인 유도 흐름을 거치도록 정리했습니다.

### 배운 점

세션 인증에서는 로그인 API 성공만으로 화면 상태가 완성되지 않습니다. 서버의 현재 사용자 정보와 프론트엔드 상태를 다시 맞추는 과정이 필요했습니다.

## 2. 조회수와 반응 수가 중복 증가하는 문제

### 문제

상세 내용을 다시 렌더링하거나 버튼을 빠르게 연속 클릭하면 조회수나 반응 수가 의도보다 많이 바뀔 수 있었습니다.

### 해결

- 조회한 모집글 ID를 세션에 기록해 같은 브라우저 세션에서는 조회수가 한 번만 증가하도록 했습니다.
- 화면 갱신 목적의 상세 조회에는 `increaseView=false`를 사용했습니다.
- 좋아요, 북마크, 모집 상태 버튼에는 처리 중 잠금과 짧은 쿨타임을 적용했습니다.
- DB에는 사용자와 모집글 기준 unique 제약을 두어 중복 반응을 막았습니다.

### 배운 점

UI에서 클릭을 막는 것도 필요하지만, 최종 데이터 일관성은 서버와 DB 제약으로 보장해야 한다는 점을 확인했습니다.

## 3. 답글 작성 후 바로 보이지 않는 문제

### 문제

답글은 저장되었지만 사용자가 현재 보고 있는 스크롤 위치에서는 바로 확인하기 어려웠습니다.

### 해결

- 답글 등록 성공 후 댓글 목록을 다시 조회했습니다.
- 새로 생성된 답글 ID를 기준으로 해당 위치까지 스크롤하고 강조 표시했습니다.
- 댓글 수정 시 기존 내용을 입력창에 먼저 채워 실수를 줄였습니다.

### 배운 점

API 성공과 사용자 체감 성공은 다릅니다. 저장 결과가 화면에서 바로 확인되는 것까지 하나의 기능 흐름으로 봐야 했습니다.

## 4. 상세 모달 스크롤이 배경 페이지까지 전달되는 문제

### 문제

모집글 상세 모달에서 스크롤을 내리거나 올릴 때 배경의 메인 페이지도 같이 움직였습니다.

### 해결

- 모달이 열려 있는 동안 문서 본문의 스크롤을 잠갔습니다.
- 상세 카드 내부에 최대 높이와 `overflow`를 적용했습니다.
- 모달을 닫을 때 기존 스크롤 위치를 복구했습니다.

### 배운 점

모달은 단순히 화면 위에 띄우는 요소가 아니라, 포커스와 스크롤 범위를 배경 화면과 분리해야 자연스럽게 동작합니다.

## 5. 관리자 화면의 데이터량과 작업 동선 문제

### 문제

회원, 모집글, 댓글을 한 화면에 모두 보여주면 데이터가 많아질수록 찾기 어렵고, 삭제 버튼까지 가로 스크롤해야 하는 문제가 있었습니다.

### 해결

- 회원, 모집글, 댓글, 신고, 작업 로그를 탭으로 분리했습니다.
- 각 영역에 검색 조건을 추가했습니다.
- 모집글은 클릭 시 미리보기 모달을 열고 그 안에서 삭제하도록 변경했습니다.
- 삭제, 제재, 신고 처리 결과는 별도 알림 UI로 보여주고 목록을 자동 갱신했습니다.
- 관리자 작업은 `admin_action_logs`에 기록했습니다.

### 배운 점

관리자 페이지는 단순히 데이터를 많이 보여주는 화면이 아니라, 찾기, 판단, 처리, 기록 확인이 이어지는 운영 흐름으로 설계해야 했습니다.

## 6. 비밀번호 재설정의 본인 확인 보강

### 문제

초기에는 이메일과 닉네임만 알면 새 비밀번호를 입력할 수 있는 구조라 실제 서비스 기준으로 약했습니다.

### 해결

- 이메일로 6자리 인증번호를 전송하는 흐름으로 바꿨습니다.
- 인증번호는 평문으로 저장하지 않고 해시로 저장했습니다.
- 인증번호 만료 시간과 실패 횟수 제한을 두었습니다.
- 인증 성공 후에는 짧은 시간 동안 한 번만 사용할 수 있는 재설정 토큰을 발급했습니다.
- 로컬 개발에서는 메일 발송 대신 서버 콘솔에 인증번호를 출력할 수 있게 분리했습니다.

### 배운 점

민감한 작업은 사용자 정보 확인과 실제 소유 확인을 구분해야 하며, 인증번호와 토큰에도 만료와 재사용 방지가 필요했습니다.

## 7. 접속 멤버가 실제 접속 상태와 다르던 문제

### 문제

초기 우측 접속 멤버 목록은 현재 사용자와 최근 댓글 작성자, 샘플 오프라인 사용자를 조합한 화면이었고 실제 온라인 상태를 나타내지는 못했습니다.

### 해결

- 세션 인증을 확인하는 WebSocket `/ws/presence`를 추가했습니다.
- 연결과 해제 시 온라인 멤버 목록을 갱신했습니다.
- 같은 사용자가 여러 탭을 열어도 사용자 단위로 중복 표시되지 않게 처리했습니다.

### 배운 점

실시간 상태는 기존 DB 데이터 조합으로 추측하기보다 연결 생명주기를 직접 관리해야 더 정확하게 표현할 수 있었습니다.
